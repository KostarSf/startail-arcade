# Client Audio System Plan

### 1. Choose audio stack and base wiring

- **Library choice**: Add a lightweight audio helper layer on top of the Web Audio API (either a tiny custom wrapper or a minimal library) to manage audio context, buffer loading, positional audio, Doppler, reverb send, and music channels.
- **New module**: Create `src/client/audio/audio-engine.ts` that initializes a single `AudioContext`, exposes methods like `playOneShot(params)`, `playLoopingMusic(id)`, `stopMusic(id, { fadeMs })`, `setCategoryVolume(category, volume)`, and attaches listener position/velocity to the camera center.
- **Client services integration**: Extend `ClientServices` in `src/client/game/types.ts` with an `audio` service (for game SFX and music) and `audioSettings` (current volumes/mutes), and wire them in `ClientEngine.#setupServices()`.

### 2. Asset loading and sound registry

- **Import sound files**: Add imports for all OGG/WAV assets in a new `src/client/audio/assets.ts` (or directly in `audio-engine.ts`), and ensure bundler loads them similarly to textures.
- **Preload buffers**: On client initialization (after Pixi and textures are ready), have the audio engine lazily decode or preload key buffers: `snd_explode`, `snd_coin`, `snd_death`, `snd_engine`, `snd_hit`, `snd_out_of_ammo`, `snd_revive`, `snd_shoot`, `snd_small_hit`, and music `msg_song`.
- **Sound metadata**: Maintain a small registry describing per-sound defaults: category (`game`, `ui`, `music`), base volume, optional reverb send level, and flags (looping, positional, Doppler-enabled).

### 3. Positional 2D audio and Doppler

- **Listener based on camera**: Each render tick (or in `CameraSystem`/`RenderSystem`), update the audio listener position and velocity from the camera center in world space, using `services.pixi.camera` and world coordinates.
- **Positional SFX API**: In the audio engine, when `playOneShot` is called with a world position `(x, y)` and optional source velocity `(vx, vy)`, create a `PannerNode` (2D mapped into Web Audio) and route through a category gain and optional reverb; compute panning and distance-based attenuation from the camera center.
- **Doppler effect**: Approximate Doppler by computing radial relative velocity between source and listener and applying a small pitch shift (playbackRate) proportional to `v_rel / c_effective` (with a tuned constant) so fast fly-bys sound shifted.

### 4. Reverb effect

- **Reverb node**: In `audio-engine.ts`, create a global convolution reverb or simple feedback reverb node on a send bus (e.g. `reverbSendGain -> convolver -> masterGain`).
- **Send control**: For each sound, use the registry’s `reverbSend` coefficient to feed some amount of its signal to the reverb bus, so explosions get more reverb than UI clicks.
- **Tuning**: Use a pre-generated IR or algorithmic reverb parameters tuned for a small-ish, spacey arena; expose a simple constant or function to tweak decay and mix if needed later.

### 5. Game event hooks for SFX

- **Network events mapping**: In `ClientEngine.#handleMessage` (or a dedicated small `audio-events` helper), hook into existing network events to trigger sounds:
- `entity:destroy` → `snd_explode` at `(x, y)`; if `score` or `playerId` correspond to the local player, also trigger `snd_coin` for score gain.
- `entity:damage` → `snd_hit` or `snd_small_hit` at `(x, y)` depending on entity type or damage amount; if this damage kills the local player, also schedule `snd_death`.
- **Player weapon and engine**: In `InputSystem` or a separate `AudioSystem`, detect when local player fires (`fireTriggered`) and play `snd_shoot` at the ship’s position; when thrust goes from off→on, start or fade in looped `snd_engine` attached to the player ship (with Doppler), and fade out when thrust stops.
- **Ammo/energy feedback**: When the client knows a shot is denied due to energy (e.g. via an existing event or a small additional server event), call `snd_out_of_ammo` as a non-positional UI/game sound centered on the listener.
- **Respawn and revive**: On successful respawn of the local player (e.g. when their ship entity appears / state changes from dead to alive), play `snd_revive`.

### 6. Music handling (`msg_song.ogg`)

- **Music channel**: Treat music as its own category routed through a `musicGain` node with its own volume and mute flag.
- **Looping & control**: Implement `playMusic("main", { loop: true, fadeInMs })` and `stopMusic("main", { fadeOutMs })` in the audio engine, using ramped gain changes.
- **Lifecycle hooks**: Start `msg_song.ogg` with fade-in when the local player is alive (e.g. when `stats.playerObject` becomes non-null or on `server:player-initialize` + first spawn), and fade out when the player dies (e.g. when `deathPosition` is set or `playerObject` becomes null because of death).

### 7. Settings, categories, and UI integration

- **Categories**: Define 3 main categories: `game` (gameplay SFX), `ui` (menu/clicks, future), `music` (background tracks), each with its own gain node.
- **Persistence**: Implement an `audio-settings` helper that reads/writes mute flags and volumes to `localStorage`, with sane defaults; load these on startup and apply to the engine.
- **Store or context**: Add a tiny `useAudioSettings` zustand store (similar to `useStats`) or extend an existing settings store, exposing `setGameMuted`, `setUIMuted`, `setMusicMuted`, and volume setters.
- **Main menu controls**: In `App.tsx` (or the main menu component once present), add controls (toggles or sliders) bound to this store and call through to the audio engine so users can disable game sounds, UI sounds, or music independently.

### 8. New ECS audio system (optional but recommended)

- **AudioSystem**: Add `src/client/game/systems/audio-system.ts` registered in the pipeline after `RenderSystem` or `EffectSystem`, which:
- Updates listener position/velocity each tick from the camera/player.
- Watches relevant ECS or `stats()` state transitions (player spawn/death, thrust changes) and fires corresponding audio engine calls, keeping `ClientEngine.#handleMessage` mostly focused on network logic.
- **Effect integration**: Optionally, have `EffectSystem` or the audio system also react to `effectQueues.explosions` (for positional explosion sounds) instead of only raw network events, ensuring SFX stay in sync with visuals.

### 9. Testing and tuning

- **Functional tests**: Verify each listed sound triggers at the correct time and position, Doppler shifts are noticeable but not extreme for fast bullets/ships, and reverb is applied mainly to explosions and large impacts.
- **Music tests**: Ensure `msg_song` loops seamlessly, starts when the player is alive, fades out on death, and respects the music mute toggle and volume.
- **Performance checks**: Confirm that audio node lifetimes are managed (stopped and disconnected after use) to avoid leaks, and that the system behaves well when sound is disabled or when the browser blocks autoplay until user interaction.
